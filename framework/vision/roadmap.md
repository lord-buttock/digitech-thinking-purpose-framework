# Roadmap

Sequenced by the two lenses: time back now first, portfolio value compounding throughout. Each project ends with something shipped and something written about it (future blog material). Revisit this document whenever a project completes or priorities shift.

## 1. Persistent planning system — NEXT

**Goal:** a lesson plan becomes a two-line request plus a five-minute review; a unit becomes a twenty-minute conversation.

Components:
- **Curriculum wiring** — largely already built: the full VC2.0 extraction (Foundation–Year 8, all learning areas, verbatim) lives in the Education vault at `Curriculum/`. The planning system points at Year 3 as the home level and at Years 2 and 4 (or further) for below/above-level differentiation, rather than duplicating any of it.
- **Class and school context** — term overview, school planning template, assessment schedule, available resources.
- **Pedagogy file** — the unified framework ([[Framework Core]]), what has/hasn't engaged the class, differentiation needs described once, CE⁵ coaching philosophy.
- **Pedagogical substrate wiring** — pointers to the Ngarri instructional model, AERO LO/SC practice, the Thinking Tools library, and the 152-skill plugin library, so every plan consults them without re-explanation.
- **CLAUDE.md** — output format matching team templates (Ngarri WALLT style, plain sections, differentiation labels), UK English, enabling/extending prompts by default.

**Done when:** three real lessons and one unit have been produced through it and the review time genuinely drops to minutes.
**Portfolio residue:** a written case study — "a persistent AI planning workspace for a classroom teacher".

## 2. Planet Builder v2

**Goal:** transform v1 from form-filling into the PTCCD testbed.

- Formation-points budget (e.g. 100) making every feature a trade-off
- 8–10 encoded consequence rules, flagged as problems to solve rather than errors
- Adversary/defence workflow — another group attempts to prove a planet non-viable; the designers' evidence-based defence is the assessable artefact
- Carry-over priority: reflect more form answers (terrain, life, aurora, civilisation) in the Babylon.js 3D model

**Done when:** one class cycle runs with the rules engine and the defence artefacts are assessable.
**Portfolio residue:** a shipped product demonstrating the framework — the strongest single proof available.

## 3. Framework completion

- Name the unified framework (candidates in [[Framework Core]]) and PTCCD
- Write the seven missing iPad category guides (Compare/Classify/Analyse next — most requested thinking type in 3–6 planning)
- Promote V2 drafts to Current; archive earlier drafts per the status-label convention
- Skills-library audit fixes already applied to the global plugins; port improvements back as needed

**Portfolio residue:** the intellectual centrepiece, publishable chapter by chapter as blog posts.

## 4. Framework database/website

The field set was designed database-ready from day one. Build the searchable planning tool (filters: year band, curriculum area, thinking category, setup complexity, artefact type). Start as a static site generated from the markdown; an Airtable-style backend only if needed.

**Portfolio residue:** a real product with real users (Ngarri staff first).

## 5. Portfolio site/blog

Built once projects 1–3 have produced content. Publishes: the framework, the planning-system case study, Planet Builder write-up, category guides. Working in public from then on. Platform decision deferred.

## 6. Design-once programmes

- **Year 6 podcast** — one good design session: crew roles, production workflow, episode pipeline.
- **Years 4–6 digital coaching offer** — artefact-based formats (screencasts, written guides, video PL) over live unscripted delivery. The framework website doubles as coaching infrastructure.
