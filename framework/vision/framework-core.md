# Framework Core

> Not more tech. Better tech.

This is the unified framework written audience-neutral, ready to lift into blog, portfolio or leadership material. It generalises two independently developed instances — Meaningful iPad Use and the PTCCD pattern for AI tasks — into one position on when technology earns its place in learning.

## The problem

Most technology integration fails in the same direction: the tool does the thinking. Students fill in digital worksheets, type brainstorms that could be sticky notes, generate AI text they cannot defend, and produce neat artefacts that contain no trace of their own reasoning. The tool was used; nothing was learned that paper would not have taught.

The conventional response is a tool-first question — *"How can we use iPads/AI in this lesson?"* — which guarantees the failure it is trying to avoid.

## The framework

### Principle 1 — Thinking first, tool second

Planning starts from the thinking purpose, never from the tool. The planning chain is:

```text
learning purpose -> thinking strategy -> student role -> technology enhancement -> artefact -> evidence of learning
```

The technology slot comes fourth, and it is optional. If the chain works without it, the lesson does not need it.

### Principle 2 — The removal test

One question decides whether the technology has earned its place:

> **If you removed the technology's output entirely, would visible student thinking remain?**

Instance forms of the same test:
- iPads: *"Why not paper?"* — every activity must have an explicit answer.
- AI: *"If you removed the AI output, would visible student thinking remain?"*

If the answer is no, the task is substitution wearing a digital costume. Redesign it or use the non-digital method.

### Principle 3 — The assessable trace

Every technology-enhanced task must leave at least one assessable trace of thinking that the technology did not produce: a recorded explanation, an annotated piece of evidence, a justified design decision, a defence of reasoning against challenge, a revision with the reason for the change.

The trace is the point. It is what the teacher assesses, what the student owns, and what proves the removal test was passed.

### Principle 4 — Name the failure mode

Every instance of the framework must articulate its own substitution danger zone — the specific weak uses that look like integration but aren't. Naming the failure mode is what makes the framework honest:

- iPad instance: digital brainstorms with no evidence; typed question lists that could be sticky notes; one confident student operating the device while others watch.
- AI instance: the tool as a form to fill out; AI output accepted as the artefact; prompts without defensible decisions behind them.

### Principle 5 — Application layer only

The framework governs the application and assessment layer of learning — the work students do *after* explicit teaching. It does not apply to the explicit teaching of foundational skills (phonics, number facts, technique), where deliberate practice and direct instruction do the work. Both instances arrived at this boundary independently; it is a load-bearing wall, not a disclaimer.

## The two instances

### Meaningful iPad Use (device layer)

Nine thinking-purpose categories act as a purpose selector (Notice/Wonder/Explore; Compare/Classify/Analyse; Investigate/Connect/Explain; …). Each category guide specifies thinking strategies, student roles, strong artefacts, an enhancement rule, a substitution danger zone, and required evidence of learning. Built database-ready for a future searchable planning tool. See [[Meaningful iPad Use - Index]].

### PTCCD (AI layer) — working draft

A design pattern for AI tasks with genuine stakes:

1. **Position** — the student holds a role with something at stake.
2. **Tension** — something forces real decisions (scarcity, time pressure, conflicting stakeholders, incomplete information, fixed rules, competing values, hostile environment).
3. **Consequence** — choices produce visible effects. This is where AI may belong: simulating outcomes fast enough for in-lesson iteration.
4. **Challenge** — the work meets resistance: a peer adversary, a real audience, a test against criteria.
5. **Defence** — the student justifies their reasoning. The defence is the assessable artefact; it is the thinking no AI produced.

Testbed: Planet Builder v2 (formation-point budget, encoded consequence rules, adversary/defence workflow).

### A third instance: the practitioner's own work

The framework also governs the designer's use of AI. A planning system built on Claude passes the removal test the same way a student task does: the professional judgement — curriculum interpretation, knowledge of the class, pedagogical decisions — remains human and visible; the AI carries re-explanation, formatting and drafting. The teacher's "assessable trace" is the five-minute review that catches what only the teacher could know.

## Naming (open)

The framework needs a name before it goes public. Working candidates:

| Candidate | Angle |
|---|---|
| **The Removal Test** | Names the framework after its sharpest tool. Memorable, slightly negative framing. |
| **Thinking Remains** | The test's answer as the name. Doubles as a quiet manifesto. |
| **Earned Technology** | Names the standard rather than the test. Leadership-friendly. |
| **Not More Tech. Better Tech.** | The existing iPad tagline, generalised. Works as a strap-line under any of the above. |

PTCCD also needs a friendlier name if it is used for coaching — candidates to be developed (e.g. something built on "stakes and defence").

## Lineage and influences

Project Zero Visible Thinking (thinking made visible as evidence), SAMR/PICRAT/TPACK/EdTech Quintet (used as checks within instances, not as the framework itself), AERO practice guides and the Victorian Teaching and Learning Model 2.0 (explicit teaching before supported application), Ron Ritchhart's Cultures of Thinking (the conditions under which any of this works).
